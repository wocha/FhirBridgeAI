# Building Intelligent Document Routers

This skill defines the architectural standard for processing massive, unlabelled PDF scan batches (z.B. gemischte Patientenakten aus dem Klinikalltag ohne Barcodes).

## Context

Hospitals frequently scan large batches of physical documents into single, multi-page PDFs. To process these automatically without manual sorting or barcode stickers, we use a 3-stage LLM-driven heuristics pipeline.

## The 3-Stage Architecture Concept

### 1. Stage: Page-by-Page OCR & Metadata Extraction

Because processing a 100-page PDF through an LLM in one go exceeds token limits and causes hallucination, the document is processed page-by-page.

- **OCR:** A tool like Tesseract or PyMuPDF extracts raw text per page.
- **Fast-LLM (Llama 3 8B):** A small, fast, local LLM extracts a lightweight JSON per page containing:
  - `patient_identifier` (Fallnummer, Name, Geburtsdatum)
  - `page_number` (if explicitly stated, e.g., "Seite 1 von 3")
  - `document_type_hint` (A rough guess, e.g., "Arztbrief", "Labor", "EKG")

### 2. Stage: Heuristic Splitter Worker

A dedicated Python worker (e.g., `document_splitter.py`) consumes the stream of page metadata and determines document boundaries.
**Splitting Logic:** A new document begins when:

- The `patient_identifier` changes.
- The `document_type_hint` changes significantly (e.g., from "Labor" to "Sturzprotokoll").
- Explicit page numbering indicates a reset (e.g., finding "Seite 1/X").
**Output:** The large PDF is spliced into N logical, single-document PDFs. These are handed to the authenticated ingestion boundary and the canonical RabbitMQ/outbox pipeline as `PENDING` jobs.

### 3. Stage: KDL Classification & Strict Extraction

The separated document (now representing a single medical report) is processed by the main `db_worker.py` / `mq_worker.py`.

- **Classifier (Routing):** The LLM compares the document text against the 383 targets in `kdl_targets.json` and returns the exact matching KDL-Code (e.g., `AD010103`).
- **Extractor:** The worker dynamically loads the exact Pydantic model generated by the KDL Night Factory (e.g., `Entlassungsbericht` from `auto_generated_kdl_models.py`) and uses `.generate_structured()` to extract the specific FHIR fields without hallucination.

## Implementation Guidelines

When instructed to implement this pipeline:

1. Ensure the `integrating-local-llms` robust retry client is used.
2. The Splitter logic must be defensive: It is better to over-split (creating two documents for the same patient) than to under-split (mixing two patients' data into one FHIR resource).
3. The KDL Classification step should ideally use vector similarity search (e.g., FAISS or ChromaDB) against the 383 descriptions to save token costs before asking the LLM to make the final choice, rather than passing all 383 options in the prompt.
